# GitHub Actions SST Deployment

## Worktree

Create branch `feat/github-actions-aws-deploy` from the current local `main` commit (`157b9dae`) in the sibling worktree `/home/josh/Projects/foosball-tracker-github-actions-aws-deploy`. The primary worktree's uncommitted `sst-env.d.ts` change is not included.

## Implementation

1. Add `.github/workflows/deploy-production.yml`.
2. Trigger the workflow on pushes to `main`, with minimal GitHub permissions: `contents: read` and `id-token: write` for AWS OIDC.
3. Serialize deployments with a non-cancelling `sst-production` concurrency group so an in-progress infrastructure update is never interrupted.
4. Set Node 22 and pnpm 11.16.0, cache pnpm using `pnpm-lock.yaml`, then install dependencies with `pnpm install --frozen-lockfile`.
5. Read `AWS_REGION` and `AWS_ROLE_TO_ASSUME` from GitHub repository variables, fail clearly when either is unset, then assume the AWS role through `aws-actions/configure-aws-credentials` using GitHub OIDC.
6. Run the web workspace's non-mutating checks before deployment: Prettier check, ESLint, TypeScript check, and Vite build via the `@foos-rank/web` filter.
7. Deploy SST's protected production stage with `pnpm exec sst deploy --stage production`. This matches the `production` retain/protect configuration in `sst.config.ts` and deploys `apps/web` through `sst.aws.TanStackStart`.
8. Include a post-assumption `aws sts get-caller-identity` diagnostic and rely on SST's successful deploy output (including the app endpoint) as deployment confirmation.

## External Prerequisites

Configure repository variables `AWS_REGION` and `AWS_ROLE_TO_ASSUME`. Create an AWS IAM GitHub Actions OIDC provider with audience `sts.amazonaws.com`; restrict the deployment role trust policy to this repository's `main` branch (`repo:OWNER/REPOSITORY:ref:refs/heads/main`) and grant only the SST deployment/bootstrap permissions necessary for the selected account and region. No long-lived AWS credentials are stored in GitHub.
