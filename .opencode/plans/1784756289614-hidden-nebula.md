# FoosRank Interactive UI Plan

## Goal

Build a polished, responsive FoosRank prototype in the existing TanStack Start app. The prototype will support one workplace league, name-only people registration, all 1v1 through 2v2 team combinations, match scoring, replayable Elo rankings, leaderboard views, and complete match history using deterministic dummy data and browser-local persistence. Backend services, accounts, invitations, and production authentication are out of scope.

## Product Decisions

- Use a clean workplace visual language: warm neutral canvas, white surfaces, navy/slate typography, restrained green brand accent, and red/blue only for team identity.
- Use the name **FoosRank** throughout metadata and UI.
- Support one active league. Initial setup creates the league and its first person; more people are added by name only.
- Keep one resumable active match at a time. Completed matches are view-only in phase one.
- Persist the prototype snapshot to `localStorage` so it survives refreshes. Include deterministic sample data and a reset action for demos.
- Treat completed match facts as canonical. Derive current ratings and per-match deltas by replaying chronological history so a future algorithm can recalculate everything.
- Use an isolated provisional Elo implementation: initial rating 1000, K-factor 32, team strength based on average member rating, no score-margin or role adjustment, and a team adjustment divided among its members. Keep full precision internally and round only for display.
- A solo player has role `both`; a two-person side requires one `attack` and one `defence` player. Roles remain match metadata and do not affect phase-one Elo.
- Scores are non-negative integers, ties are invalid, and no first-to-10/win-by-two rule is assumed.

## Architecture And Data

1. Create domain models under `apps/web/src/domain/` for `League`, `Person`, active and completed matches, participants, team colors, positions, scores, and the versioned app snapshot.
2. Store every completed match with stable ID, chronological sequence, format, participant IDs, red/blue assignment, position, final score, timestamps, and rating algorithm version. Do not use a person's current Elo as the source of truth.
3. Implement pure modules for:
   - Match/roster validation and normalized duplicate-name checks.
   - Chronological Elo replay and per-match before/after deltas.
   - Leaderboard, win/loss, win rate, streak, recent-form, and dashboard selectors.
4. Add `apps/web/src/data/foosrank-repository.ts` as an asynchronous browser repository over one atomic, schema-versioned `localStorage` snapshot. It will safely initialize empty data, tolerate malformed/unsupported stored data, create IDs with `crypto.randomUUID()`, and expose setup, add-person, start/cancel/complete-match, load-demo, and reset operations.
5. Add deterministic seed data in `apps/web/src/data/demo-state.ts`, including enough people and historical matches to exercise 1v1, 2v1, 1v2, and 2v2 layouts.
6. Add Query keys, queries, mutations, and feature-facing hooks under `apps/web/src/data/`. Use one canonical state query; successful mutations update that cached snapshot immediately. TanStack Query is the component data boundary, while the repository remains the persistence boundary.
7. Refactor `apps/web/src/providers/TanstackQueryProvider.tsx` so each application instance owns a stable `QueryClient` with sensible local-data defaults instead of sharing a process-global SSR cache.

## Routes And Flows

Use TanStack Router file routes and a pathless league layout:

- `/`: client-safe entry that sends an initialized browser to setup or dashboard without reading `localStorage` during SSR.
- `/setup`: league name plus first person name, with a secondary “Load sample league” action and browser-local-data notice.
- `/overview`: league summary, top rankings, recent matches, and prominent start/resume-match action.
- `/people`: inline name-only registration and roster cards/table with rating and record.
- `/matches/new`: a focused match flow covering format, teams/positions, review/start, score entry, and result summary.
- `/matches`: complete reverse-chronological match history.
- `/matches/$matchId`: auditable result details, player positions, score, and replay-derived Elo changes.
- `/leaderboard`: full ranking with Elo, played, wins/losses, win rate, streak, and recent form.

The pathless league route will render the responsive app shell and guard league-only pages after client state initializes. It redirects empty state to setup and avoids hydration mismatch. `apps/web/src/routeTree.gen.ts` remains generated and will be regenerated, never manually edited.

## Match Interaction

1. Choose `1v1`, `2v1`, `1v2`, or `2v2`.
2. Fill fixed red and blue player slots from active people. Prevent duplicate selections across all slots.
3. Enforce `both` for solo sides and attack/defence coverage for doubles sides; provide a clear team-swap action.
4. Review and start the match, persisting its start timestamp and participant metadata.
5. Show large touch-friendly red and blue score controls. Reject blank, negative, fractional, excessive, or tied scores.
6. Save once while the mutation is pending, append the immutable completed match, clear the active match, replay Elo, and show winner plus rating deltas.
7. If an active match already exists, offer resume or explicit cancel rather than replacing it silently.

## UI Structure

- Add thin, local primitives in `apps/web/src/components/ui/` for buttons, fields, inputs/selects, cards, badges, initials avatars, stat tiles, empty/loading/error states, and confirmation UI. Do not add a general component framework.
- Add the app shell and navigation in `apps/web/src/components/`: desktop sidebar, compact tablet treatment, mobile bottom navigation, page headers, and FoosRank mark.
- Organize screen-specific components under `apps/web/src/features/setup/`, `people/`, `overview/`, `matches/`, and `leaderboard/`.
- Update `apps/web/src/styles.css` with Tailwind v4 CSS-first theme tokens, base typography, semantic surfaces/borders/focus rings, red/blue team tokens, tabular numeric styling, safe-area handling, and reduced-motion behavior.
- Update `apps/web/src/routes/__root.tsx` metadata for FoosRank while preserving `HeadContent`, `Scripts`, and existing document plumbing. Keep devtools unobtrusive in development.
- Replace the starter screen in `apps/web/src/routes/index.tsx` with the client-safe entry flow.

## Responsive And Accessibility

- Use semantic landmarks, one visible `h1` per page, a skip link, and typed router links with clear active states.
- Use `fieldset`/`legend` for format and team choices, persistent labels for every control, connected inline errors, and live regions for mutation results.
- Never communicate teams or outcomes by color alone; always pair red/blue with text and clear labels.
- Maintain visible `:focus-visible` styles, keyboard operation, 44px minimum touch targets, dialog focus management, and focus movement between match steps.
- Stack team builders on narrow screens, use purpose-built mobile leaderboard/history cards instead of clipped tables, keep score entry usable beside the table, and reserve space for mobile navigation safe areas.

## Validation And Tests

1. Add `typecheck`, Vitest, and test scripts to `apps/web/package.json`; add Testing Library only for a small set of valuable interaction tests.
2. Unit test Elo replay for equal players, upsets, all asymmetric/symmetric team sizes, zero-sum adjustment, ordering, repeatability, and display-only rounding.
3. Unit test names, team sizes, duplicate participants, role coverage, score rules, and selectors.
4. Test repository initialization, persistence/reload, corrupt data fallback, active-match resume, completion, demo seed, and reset.
5. Add focused interaction coverage for setup, duplicate person rejection, each match format, invalid score prevention, completed-match history, and immediate leaderboard updates.

## Implementation Order

1. Domain types, validation, replayable Elo, selectors, and unit tests.
2. Versioned local repository, sample state, and Query integration.
3. Setup route, state-aware entry/league guard, and responsive app shell.
4. People registration and roster.
5. Match builder, active match, score entry, completion, and result summary.
6. Overview, leaderboard, history, and match detail.
7. Responsive/accessibility polish, interaction tests, route generation, and full verification.
